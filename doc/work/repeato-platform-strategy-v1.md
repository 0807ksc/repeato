# Repeato Platform Strategy v1 (Case 1 vs Case 2)

## 메타
- 버전: v1
- 작성일: 2026-02-22
- 상태: Active
- 목적: 범용 암기 앱 전략 확정

## 전제 (핵심 원칙)
- Repeato의 본질은 "영어 앱"이 아니라 "무엇이든 암기를 돕는 플랫폼"이다.
- 콘텐츠(영단어/천자문/기타)는 교체 가능한 모듈로 다뤄야 한다.

## 비교 대상
### Case 1
- 단일 앱 + 인앱 콘텐츠 구매/활성화
- 예: 중학교 단어, 천자문을 인앱으로 구매 후 학습장에 추가

### Case 2
- 콘텐츠별 별도 앱으로 빌드/출시
- 예: 중학영단어 앱, 천자문 앱을 각각 별도 앱으로 등록

## 평가 기준
1. 제품 본질 일치도(범용 암기 플랫폼 유지)
2. 개발/운영 복잡도
3. 출시 속도
4. 수익/가격 전략 유연성
5. 사용자 경험 연속성
6. 마케팅/스토어 확장성

## 장단점
### Case 1 장점
- 범용 플랫폼 철학과 가장 일치
- 코드베이스 1개로 유지보수 비용이 낮음
- 사용자 데이터/학습 히스토리 통합 관리 가능
- 교차 판매(영단어 사용자에게 천자문 제안) 쉬움
- 기능 개선 시 전체 사용자에게 즉시 반영 가능

### Case 1 단점
- 앱 내부 정보구조가 복잡해질 수 있음
- 초기 콘텐츠 권한/결제/라이선스 구조 설계 필요
- 특정 콘텐츠 키워드로 스토어 검색 노출은 불리할 수 있음

### Case 2 장점
- 각 앱이 특정 키워드에 최적화되어 마케팅이 쉬움
- 사용자에게 목적이 선명한 앱 경험 제공 가능
- 가격/브랜딩을 앱별로 다르게 가져가기 용이

### Case 2 단점
- 코드/배포/QA/운영이 앱 수만큼 증가
- 기능 개선/버그 수정 동시 반영 비용 큼
- 사용자 학습 데이터가 앱 간 분절됨
- 장기적으로 플랫폼 정체성이 약해질 위험

## 전 Agent 회의 요약
- PM: 장기 제품 정체성은 Case 1이 더 명확
- Architect: Case 2는 운영 복잡도와 테스트 비용이 크게 증가
- Designer: 사용자 입장에서 학습 자산 통합(한 앱)이 더 자연스러움
- Backend/Data: 학습 데이터 통합/개선 루프는 Case 1이 유리
- QA/DevOps: 다중 앱 운영은 릴리즈 리스크 증가
- Education/Cognitive Science: 콘텐츠 확장 실험은 Case 1 구조가 반복 학습 정책 유지에 유리

## 권고안 (최종)
- 1순위 권고: **Case 1 채택** (단일 앱 + 인앱 콘텐츠 모델)
- 단, 향후 마케팅 실험용으로 Case 2를 파생할 수 있도록 아키텍처는 "콘텐츠 패키지 분리형"으로 설계한다.

## 실행 전략
### Phase 1 (지금)
- Case 1로 MVP 개발
- 콘텐츠 패키지 구조 설계:
  - 패키지 메타(이름/난이도/태그)
  - 카드 데이터
  - 과금/권한 정보

### Phase 2 (확장)
- 브랜드/마케팅 필요 시 Case 2 형태의 파생 앱 자동 생성 가능 구조 준비
- 공통 코어 + 콘텐츠 번들만 교체 가능한 빌드 파이프라인 검토

## 아키텍처 원칙
1. Core Engine: 복습 스케줄/학습 세션/분석은 공통
2. Content Pack: 데이터와 카탈로그는 분리
3. Commerce Layer: 패키지별 구매/활성화
4. Build Profile: 필요 시 콘텐츠 번들 기반 멀티앱 빌드 가능

## 결론 한 줄
- 지속 가능성과 제품 본질을 기준으로 현재는 **Case 1이 최적**이고,
- 미래 확장성을 위해 기술 구조는 **Case 2 파생 가능성**을 열어두는 방식이 가장 합리적이다.
