# Repeato Service Open Plan v1

## 메타
- 버전: v1
- 작성일: 2026-03-08
- 상태: Active
- 목적: PM이 각 조직 작업을 조율한 결과, 서비스 오픈까지의 실행 리스트를 정의한다.

## 1. PM 조율 원칙
- 기획 문서 확정 없이 구현하지 않는다.
- 로컬 MVP를 먼저 완성한다.
- 동기화/결제는 설계만 유지하고 실제 구현은 후속 단계로 미룬다.
- 서버리스가 꼭 필요한 시점은 PM이 1차 판단하고, 불명확하면 사용자에게 의사결정을 요청한다.

## 2. 조직별 책임
- PM: 우선순위, blocked 해제, 최종 범위 관리
- Designer: 탭별 상태 화면과 문구 확정
- Architect: 도메인/SQLite/상태 모델 고정
- Frontend Engineer: Flutter 구현과 로컬 저장 연동
- Backend Engineer: 현재는 설계 검토만 수행
- QA Engineer: 마스터 시나리오와 회귀 테스트 유지
- Data Analyst: KPI 정의와 이벤트 스키마 확정
- Security/Privacy: 동기화/로컬 저장 개인정보 기준 검토

## 3. 오픈 전 작업 리스트

### Phase 0. 기획/설계 잠금
1. 탭 간 충돌 조정안 확정
2. 카드 템플릿 확정
3. 세션 규칙/완료 기준 확정
4. KPI 5개와 v1 노출 KPI 3개 확정
5. SQLite/도메인 모델 확정

### Phase 1. 로컬 MVP 구현
1. `main.dart` 단일 파일 구조 분리
2. SQLite 저장 계층 도입
3. Today 3단계 평가/완료 상태 구현
4. Decks 목록/상세/일시중지 구현
5. Add 직접 입력 구현
6. Insights 핵심 KPI 3개 구현
7. Profile 로컬 설정 구현

### Phase 2. 품질 기준 확보
1. Widget 테스트 확장
2. 상태 스냅샷 테스트 추가
3. 앱 재실행 후 상태 복원 테스트
4. 전체 QA 마스터 시나리오 재검증
5. 접근성/빈 상태/오류 상태 검토

### Phase 3. 베타 오픈 준비
1. 앱 아이콘/문구/브랜딩 정리
2. 샘플 콘텐츠 품질 검수
3. 로컬 데이터 마이그레이션 정책 정리
4. 개인정보/동기화 안내 문구 초안 작성
5. TestFlight/내부 배포 기준 정의

### Phase 4. 후속 확장 계획
1. Google Drive 수동 동기화 MVP 설계 검증
2. 결제/체험 정책 문서 확정
3. 서버리스 필요성 재판단

## 4. 현재 blocked 항목
- Insights:
  - KPI/이벤트 스키마 미확정
- Profile:
  - 동기화/결제는 기획안만 있고 실제 구현 범위 제외
- Add 대량 가져오기:
  - 템플릿/중복 정책 미확정

## 5. 지금 바로 할 일
1. `Today + Add + Decks` 공통 상태 키 확정
2. SQLite 스키마를 문서에서 코드 설계로 내리기
3. `main.dart` 분리 설계
4. QA가 로컬 저장 이후 회귀 시나리오를 추가

## 6. 오픈 기준
- MVP-1 오픈 기준:
  - Today/Decks/Add/Insights/Profile 기본 흐름 동작
  - SQLite 기반 로컬 저장/복원 동작
  - 핵심 QA 시나리오 P0/P1 통과
  - `flutter analyze`, `flutter test` 통과

## 7. 사용자 의사결정이 필요한 시점
- 서버리스 백엔드가 필요하다고 PM이 1차 판단했을 때
- 결제 검증/복원 신뢰도를 서버 없이 감당하기 어렵다고 판단될 때
- Google Drive만으로 다중 디바이스 충돌 해결 UX가 과도하게 복잡해질 때
