## 1. 작업 프레임
- 기본 Agent 세트: `PM + Product Designer + Technical Architect + Frontend Engineer + QA Engineer`
- 필요시 추가 Agent: `Backend Engineer`
- 추가 이유: 덱별 목표, 활성/중지 상태, 통계 집계 기준이 서버 동기화 모델과 연결될 수 있기 때문

## 2. 참여자 논의 정리
- PM: `Decks`는 "정리 화면"이지 학습보다 앞에 오면 안 된다.
- UX 디자이너: 덱 수가 많아져도 사용자가 길을 잃지 않게 상태와 우선순위가 보여야 한다.
- App 개발자: 덱 상태(`진행 중/일시중지/완료`)와 목표 수치가 분리돼야 구현이 단순하다.
- 사용자: 내가 지금 어떤 학습 과제가 살아 있는지, 다음 복습이 언제인지 빠르게 알고 싶다.

## 3. 사용자 확인 질문
- 덱 목록에서 가장 먼저 보고 싶은 정보는 무엇인가:
  - 다음 복습 시각
  - 진행률
  - 남은 카드 수
- 덱이 많을 때 카테고리보다 `지금 해야 함` 정렬이 더 유용한가?
- 덱 상세에서 바로 `학습 시작`하는 버튼이 항상 고정돼야 하는가?

## 4. 상세 사양

### 4.1 제품 목적
- 사용자가 학습 과제를 추가, 정리, 우선순위 조정하고 바로 `Today`로 연결되게 한다.

### 4.2 핵심 화면
1. Deck List
2. Deck Detail
3. Deck Edit

### 4.3 Deck List
- 최상단은 `오늘 복습 있음` 덱을 우선 노출한다.
- 각 카드에는 덱 이름, 남은 카드 수, 다음 복습 시각, 상태 배지를 표시한다.
- 빈 상태에서는 `Add에서 첫 덱 만들기` CTA를 노출한다.

### 4.4 Deck Detail
- 노출 정보: 카드 수, 숙련도 분포, 다음 복습 예정, 일일 목표
- CTA 우선순위:
  - `학습 시작`
  - `카드 관리`
  - `설정`

### 4.5 상태 정책
- `진행 중`: Today 큐에 포함
- `일시중지`: Today 큐에서 제외
- `완료`: 신규 카드 0, 복습 0 상태일 때 자동 후보

## 5. 연동 질문
- `Today` Agent에게 질문: 덱별 대기량을 Today 홈에서 몇 개까지 보여줄 것인가?
- `Add` Agent에게 질문: 새 카드 추가 후 Deck Detail 갱신 시점을 즉시 반영으로 둘 것인가?
- `Insights` Agent에게 질문: 덱별 지표와 전체 지표의 우선 노출 기준은 무엇인가?
- `Backend Engineer` 확인 필요:
  - 덱별 목표/상태가 기기 간 동기화 대상인지
  - 덱 삭제 시 카드/기록 보존 정책이 무엇인지

## 6. 3차 검토 요약
1. 1차 검토
- 목적: 학습 과제 관리 허브
- 금지: 복잡한 관리자형 UI

2. 2차 검토
- 합의: `Today` 우선 정렬과 상태 배지 도입
- 보류: 폴더/태그 다단계 구조는 v1 제외

3. 3차 검토
- 합의: `Decks`는 구조 정리와 빠른 재진입에 집중
- 합의: 상세 화면에는 `학습 시작` CTA를 항상 유지

## 7. 상위 의사결정자 리드백 2회
1. 1차 리드백 반영
- 목록 정보량을 줄이기 위해 숙련도 분포는 상세 화면으로 내렸다.

2. 2차 리드백 반영
- `진행 중/일시중지/완료` 3상태만 우선 채택했다.

## 8. 최종 결정
- `Decks`는 학습 구조 정리와 재진입 허브로 정의한다.
- 목록은 `오늘 해야 할 덱` 우선, 상세는 `학습 시작` 우선으로 설계한다.

## 9. 후속 작업
1. 덱 상태와 동기화 정책을 서버 범위와 함께 확정한다.
2. `Today`와 덱 대기량 표기 규칙을 맞춘다.
