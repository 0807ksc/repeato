# Repeato GitHub Tracking Workflow v1

## 메타
- 버전: v1
- 작성일: 2026-03-08
- 상태: Active
- 목적: 개발 작업은 반드시 `이슈 -> 브랜치 -> 구현 -> PR -> QA -> 머지` 순서로 추적되게 한다.

## 1. 강제 원칙
- 개발 작업은 GitHub 이슈 없이 시작하지 않는다.
- 각 구현 작업은 개발 리더 포함 3명의 개발자가 역할을 나눠 구현 방안을 먼저 검토한다.
- 검토 결과는 GitHub 이슈 본문 또는 코멘트에 기록한다.
- 브랜치는 반드시 이슈 번호를 포함해 생성한다.
- 작업 완료 후 PR을 생성하고, QA/PM 검토 후 머지한다.
- 머지 전에는 `main` 직접 커밋을 금지한다. 단, 문서성 긴급 수정은 PM 승인 필요.
- 개발 작업은 테스트 라인 커버리지 70% 이상 유지를 기본 원칙으로 삼는다.
- 커버리지가 70% 미만이면 PR에 측정값, 원인, 보완 일정, 승인자를 남긴다.

## 2. 개발 조직 3인 구성

### 기본 3인
1. Development Lead
- 역할:
  - 구현 방안 리뷰 주관
  - 이슈 범위/브랜치/PR 제목 결정
  - 최종 구현 방향 승인

2. Feature Engineer
- 역할:
  - 실제 기능 구현
  - 테스트 추가
  - PR 본문 초안 작성

3. Integration Engineer
- 역할:
  - 기존 구조/다른 탭/데이터/빌드 영향 검토
  - 회귀 리스크 식별
  - QA 인수인계 포인트 정리

### 조건부 추가
- Data-heavy task: `Data Analyst` 또는 `Backend Engineer`
- Security-sensitive task: `Security/Privacy`

## 3. 작업 시작 절차
1. PM이 작업 태그/담당 조직을 지정한다.
2. Development Lead가 3인 역할을 배정한다.
3. 3인이 구현 방안을 짧게 합의한다.
4. 테스트 전략과 예상 커버리지 영향(기본 70% 유지)을 이슈에 기록한다.
5. GitHub 이슈를 생성한다.
6. 브랜치명을 `feat/<issue-number>-<slug>` 또는 `fix/<issue-number>-<slug>`로 만든다.
7. 브랜치에서만 작업한다.

## 4. 이슈 템플릿 필수 항목
- 배경
- 목표
- 범위
- 비범위
- 구현 방안 검토 결과
- 개발 3인 역할 분담
- QA 관점 체크포인트
- 테스트 전략 / 커버리지 계획
- 사용자 피드백 필요 여부

## 5. PR 템플릿 필수 항목
- 연결 이슈
- 변경 요약
- 테스트 결과
- 커버리지 결과
- 사용자 영향
- QA 확인 포인트
- PM 의사결정 필요 항목

## 6. 머지 게이트
- Gate 1: Development Lead 자체 검토
- Gate 2: QA 테스트 통과 및 커버리지 70% 유지 확인
- Gate 3: PM 범위/기획 적합성 확인
- Gate 4: main 머지

## 7. 현재 적용 상태
- GitHub 원격은 연결돼 있음
- `gh auth status` 결과 토큰 무효로 실제 이슈/PR 생성은 현재 blocked
- 인증 복구 후 바로 적용 가능

## 8. 인증 blocked 시 임시 규칙
- `doc/context-log.md`에 `GitHub issue pending` 상태를 기록한다.
- 실제 구현 시작 전 또는 최소 PR 전에는 GitHub 이슈를 복구 생성한다.
- 인증 전까지는 로컬 브랜치 생성과 문서 준비까지만 허용한다.
